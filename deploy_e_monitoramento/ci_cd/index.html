    CI/CD (Integração e Entrega Contínua)

    Atualmente, o projeto SisCoE não possui um pipeline de CI/CD formalmente implementado. Esta seção descreve uma proposta de fluxo de trabalho de CI/CD utilizando GitHub Actions, que pode ser adotada para automatizar os processos de teste e deploy, melhorando a qualidade e a velocidade das entregas.

    Objetivos do CI/CD

    • Integração Contínua (CI): Automatizar a execução de testes e a verificação da qualidade do código a cada novo commit, garantindo que novas alterações não quebrem a funcionalidade existente.
    • Entrega Contínua (CD): Automatizar o processo de deploy da aplicação em um ambiente de homologação ou produção após a passagem bem-sucedida pelo pipeline de CI.

    Pipeline de Integração Contínua (CI)

    Este pipeline seria acionado a cada push para as branches main ou develop, ou a cada abertura de Pull Request.

    Passos: 1. Checkout do Código: O workflow do GitHub Actions faz o checkout da versão mais recente do código. 2. Build da Imagem Docker: A imagem da aplicação Django é construída utilizando o Dockerfile e o docker-compose. Isso garante que o ambiente de teste seja idêntico ao de produção. 3. Execução dos Testes: O comando python manage.py test é executado dentro do contêiner da aplicação para rodar todos os testes unitários e de integração. 4. Análise de Código (Opcional): Ferramentas de análise estática de código (linting), como Flake8 ou Ruff, podem ser executadas para garantir a conformidade com as convenções de código. 5. Push da Imagem (em caso de sucesso): Se todos os testes passarem, a imagem Docker construída é tagueada com a versão do commit e enviada para um registro de contêineres, como o Docker Hub ou o GitHub Container Registry.

    Pipeline de Entrega Contínua (CD)

    Este pipeline seria acionado manualmente ou automaticamente após a conclusão bem-sucedida do pipeline de CI em uma branch de produção (como a main).

    Passos: 1. Conexão com o Servidor: O workflow se conecta ao servidor de produção via SSH. 2. Pull da Nova Imagem: O servidor de produção baixa a nova versão da imagem da aplicação a partir do registro de contêineres. 3. Atualização da Aplicação: O docker-compose é utilizado para reiniciar os serviços, utilizando a nova imagem. 

    docker-compose pull app
docker-compose up -d --no-deps app
    4. Execução de Migrações: O comando python manage.py migrate é executado no novo contêiner da aplicação para aplicar quaisquer migrações de banco de dados pendentes.

    A implementação deste pipeline de CI/CD traria mais segurança e agilidade ao processo de desenvolvimento e deploy do SisCoE.
