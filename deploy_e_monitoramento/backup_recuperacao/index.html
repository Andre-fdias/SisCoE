    Backup e Recuperação

    Uma estratégia de backup e recuperação de desastres é vital para garantir a integridade e a disponibilidade dos dados do SisCoE. Esta seção descreve a persistência de dados atual e recomenda um plano de backup robusto.

    Persistência de Dados Atual

    Atualmente, a persistência dos dados do banco de dados é garantida pelo uso de um volume Docker nomeado (pgdata) para o serviço do PostgreSQL, conforme definido no docker-compose.yml.

    services:
  db:
    # ...
    volumes:
      - pgdata:/var/lib/postgresql/data
# ...
volumes:
  pgdata:

    Isso garante que os dados do banco de dados sobrevivam à recriação do contêiner, mas não protege contra a corrupção de dados, exclusão acidental ou falha do host. Portanto, não é uma estratégia de backup completa.

    Estratégia de Backup Recomendada

    1. Backup do Banco de Dados (PostgreSQL)

    A abordagem mais comum e recomendada para fazer backup de um banco de dados PostgreSQL é utilizar o utilitário pg_dump. Este comando pode ser executado periodicamente através de um script e um cron job no servidor host.

    Script de Backup (backup.sh):

    #!/bin/bash

# Variáveis
DB_CONTAINER_NAME="siscoe.db"
DB_USER="postgres"
DB_NAME="siscoe_db"
BACKUP_DIR="/path/to/backups"
DATE=$(date +%Y-%m-%d_%H-%M-%S)
BACKUP_FILE="$BACKUP_DIR/siscoe_db_backup_$DATE.sql.gz"

# Cria o diretório de backup se ele não existir
mkdir -p $BACKUP_DIR

# Executa o pg_dump dentro do contêiner e comprime a saída
docker exec $DB_CONTAINER_NAME pg_dump -U $DB_USER -d $DB_NAME | gzip > $BACKUP_FILE

# (Opcional) Remove backups mais antigos que 7 dias
find $BACKUP_DIR -type f -name "*.sql.gz" -mtime +7 -delete

    Este script deve ser agendado para rodar diariamente (ou com a frequência desejada) usando um cron job.

    2. Backup de Arquivos de Mídia

    Os arquivos enviados pelos usuários (como fotos de perfil) são armazenados no diretório media/, conforme definido em settings.py. Este diretório também precisa ser incluído na rotina de backup.

    Um comando rsync ou tar pode ser usado para arquivar e comprimir este diretório periodicamente.

    tar -czf /path/to/backups/siscoe_media_$DATE.tar.gz /path/to/project/media/

    Estratégia de Recuperação

    1. Recuperação do Banco de Dados

    Para restaurar um backup do banco de dados, o processo inverso é realizado:

    1. Copie o arquivo de backup para o servidor, se necessário.
    2. Crie um banco de dados vazio no contêiner do PostgreSQL.
    3. Importe o backup usando o utilitário psql.

    Script de Restauração (restore.sh):

    #!/bin/bash

# Variáveis
DB_CONTAINER_NAME="siscoe.db"
DB_USER="postgres"
DB_NAME="siscoe_db"
BACKUP_FILE="/path/to/backup/to/restore.sql.gz"

# Comando de restauração
gunzip < $BACKUP_FILE | docker exec -i $DB_CONTAINER_NAME psql -U $DB_USER -d $DB_NAME

    2. Recuperação dos Arquivos de Mídia

    Basta descompactar o arquivo de backup do diretório media/ no local apropriado.

    tar -xzf /path/to/backup/siscoe_media.tar.gz -C /path/to/project/

    A implementação de uma rotina de backup automatizada e testada regularmente é um passo crítico para a estabilidade e segurança do sistema em produção.
