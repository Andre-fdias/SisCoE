    Boas Práticas de Segurança

    A segurança é um aspecto fundamental no desenvolvimento do SisCoE. O projeto adota várias boas práticas e aproveita as funcionalidades de segurança integradas do framework Django para proteger a aplicação e seus dados.

    Proteções Nativas do Django

    O Django oferece, por padrão, proteção contra várias das vulnerabilidades mais comuns da web:

    • Prevenção de SQL Injection: O ORM do Django utiliza consultas parametrizadas, o que previne a grande maioria dos ataques de injeção de SQL. Todo o acesso ao banco de dados através do ORM é, por padrão, seguro contra esta vulnerabilidade.

    • Proteção contra Cross-Site Scripting (XSS): O sistema de templates do Django escapa automaticamente todas as variáveis que são renderizadas no HTML. Isso significa que qualquer código malicioso inserido por um usuário nos dados (ex: um script em um campo de texto) será renderizado como texto plano e não será executado no navegador de outros usuários.

    • Proteção contra Cross-Site Request Forgery (CSRF): Todos os formulários que utilizam o método POST são protegidos com um token de CSRF ({% csrf_token %}). O Django verifica a validade deste token a cada requisição POST, garantindo que a requisição tenha se originado de um formulário legítimo da própria aplicação, e não de um site malicioso.

    • Prevenção de Clickjacking: O django.middleware.clickjacking.XFrameOptionsMiddleware está habilitado, o que impede que o site seja exibido dentro de um <iframe> em outro site, prevenindo ataques de clickjacking.

    Gerenciamento de Segredos

    Segredos, como a SECRET_KEY do Django e chaves de API, nunca devem ser versionados no código-fonte.

    • python-decouple: O projeto utiliza a biblioteca python-decouple para carregar essas informações sensíveis a partir de variáveis de ambiente ou de um arquivo .env (que deve ser incluído no .gitignore).

    Exemplo (settings.py): 

    from decouple import config

SECRET_KEY = config('SECRET_KEY')
DEBUG = config('DEBUG', default=False, cast=bool)

    Configurações de Produção

    Para o ambiente de produção, as seguintes configurações de segurança são cruciais:

    • DEBUG = False: Em modo de debug, o Django exibe páginas de erro detalhadas que podem vazar informações sensíveis sobre a configuração do projeto. Em produção, o DEBUG deve sempre ser False.

    • ALLOWED_HOSTS: Esta configuração deve ser preenchida com o domínio exato da aplicação para prevenir ataques de envenenamento de cabeçalho Host.

    • HTTPS: A aplicação deve ser servida exclusivamente sobre HTTPS para criptografar o tráfego entre o cliente e o servidor. O Nginx no docker-compose.yml pode ser configurado para redirecionar todo o tráfego HTTP para HTTPS.

    Manutenção de Dependências

    Manter as dependências (tanto do Python quanto do Node.js) atualizadas é vital para a segurança. Versões desatualizadas podem conter vulnerabilidades conhecidas. É recomendado utilizar ferramentas como o Dependabot do GitHub para monitorar e criar Pull Requests para atualizar as dependências automaticamente.
