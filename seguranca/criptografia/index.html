    Criptografia

    A criptografia é uma camada de defesa essencial para proteger a confidencialidade e a integridade dos dados no SisCoE. A estratégia de criptografia abrange os dados em trânsito e em repouso.

    Criptografia em Trânsito

    Toda a comunicação entre o navegador do usuário e o servidor da aplicação deve ser protegida com HTTPS (HTTP sobre TLS).

    • Implementação: Em um ambiente de produção, a terminação TLS/SSL é geralmente responsabilidade do proxy reverso (Nginx). O Nginx é configurado com um certificado SSL/TLS (obtido de uma Autoridade Certificadora como Let's Encrypt) para criptografar todo o tráfego que chega na porta 443 (HTTPS).
    • Fluxo: O tráfego entre o usuário e o Nginx é criptografado. A comunicação interna entre o Nginx e a aplicação Django (Gunicorn) ocorre dentro da rede privada do Docker e pode ou não ser criptografada, pois acontece em um ambiente controlado.

    A configuração para forçar o redirecionamento de todo o tráfego HTTP para HTTPS no Nginx é uma prática de segurança padrão e recomendada.

    Criptografia de Senhas

    As senhas dos usuários nunca são armazenadas em texto plano. O Django gerencia a segurança das senhas de forma robusta.

    • Hashing de Senhas: Em vez de armazenar a senha, o Django armazena um "hash" da senha. Um hash é o resultado de uma função criptográfica de mão única que é computacionalmente difícil de reverter.
    • Algoritmo Utilizado: O projeto está configurado para usar o Argon2, através da biblioteca argon2-cffi. O Argon2 é atualmente um dos algoritmos de hashing mais recomendados para senhas, pois é resistente a ataques de força bruta tanto em CPUs quanto em GPUs.

    Quando um usuário faz login, a senha fornecida é passada pela mesma função de hashing, e o resultado é comparado com o hash armazenado no banco de dados. Se os hashes corresponderem, a senha está correta.

    Criptografia de Dados em Repouso (At Rest)

    Refere-se à criptografia dos dados enquanto estão armazenados no disco do servidor.

    • Nível de Aplicação: Atualmente, o projeto não parece implementar criptografia em nível de aplicação para campos específicos do banco de dados (ex: criptografar o CPF de um militar antes de salvá-lo). Se dados extremamente sensíveis forem armazenados no futuro, o uso da biblioteca django-cryptography pode ser considerado para criptografar campos individuais.

    • Nível de Banco de Dados/Sistema de Arquivos: A criptografia dos dados em repouso é mais comumente tratada na camada de infraestrutura.

      • PostgreSQL: O PostgreSQL oferece mecanismos para criptografar dados em colunas específicas ou o banco de dados inteiro.
      • Criptografia de Disco: O provedor de nuvem ou o sistema operacional do servidor host pode ser configurado para criptografar todo o disco onde os dados do banco de dados e os arquivos de mídia estão armazenados. Esta é uma prática de segurança recomendada para proteger os dados em caso de acesso físico não autorizado ao servidor.
