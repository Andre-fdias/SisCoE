    Controle de Acesso

    O controle de acesso no SisCoE é implementado através de um sistema de permissões hierárquicas, construído sobre o sistema de autenticação do Django. Isso garante que os usuários só possam acessar as funcionalidades e os dados para os quais eles têm autorização.

    Modelo de Usuário Customizado

    O projeto utiliza um modelo de usuário customizado (accounts.User), que estende o AbstractBaseUser do Django. Este modelo inclui um campo permissoes que define o nível de acesso do usuário no sistema.

    Modelo User (accounts/models.py): 

    class User(AbstractBaseUser, PermissionsMixin):
    # ... outros campos ...

    PERMISSOES_CHOICES = (
        ("basico", "Básico"),
        ("visitantes", "Visitante"),
        ("sgb", "SGB"),
        ("gestor", "Gestor"),
        ("admin", "Admin"),
    )
    permissoes = models.CharField(
        _('Permission Level'),
        max_length=20,
        choices=PERMISSOES_CHOICES,
        default="basico"
    )

    # ...

    Níveis de Permissão

    Os níveis de permissão são hierárquicos, o que significa que um nível mais alto herda todas as permissões dos níveis inferiores.

    1. Básico: O nível mais baixo, com acesso apenas a funcionalidades essenciais e aos seus próprios dados.
    2. Visitante: Um nível intermediário com permissões de visualização adicionais.
    3. SGB: Permite a gestão de dados relacionados a um SGB (Subgrupamento de Bombeiros) específico.
    4. Gestor: Acesso de gerenciamento amplo sobre os dados da aplicação, mas sem permissões de superusuário.
    5. Admin: Acesso de administrador ao painel do Django e a funcionalidades de gerenciamento do sistema.
    6. Superuser: Acesso irrestrito a todo o sistema, concedido pela flag is_superuser.

    Verificação de Permissões

    A verificação do nível de permissão de um usuário é centralizada no método has_permission_level() do modelo User.

    def has_permission_level(self, required_level):
    hierarchy = {
        'basico': 0,
        'visitantes': 1,
        'sgb': 2,
        'gestor': 3,
        'admin': 4
    }

    if self.is_superuser:
        return True

    user_level = hierarchy.get(self.permissoes)
    req_level = hierarchy.get(required_level)

    if None in (user_level, req_level):
        return False

    return user_level >= req_level

    Este método é então utilizado em views e middlewares para restringir o acesso.

    Protegendo as Views

    O acesso às views é protegido utilizando uma combinação de decorators do Django e decorators customizados.

    • @login_required: O decorator padrão do Django que garante que apenas usuários autenticados possam acessar uma determinada view.

    • Decorators Customizados: O projeto possui decorators customizados (em accounts/decorators.py) que verificam o nível de permissão.

    Exemplo de uso em uma view: 

    from django.contrib.auth.decorators import login_required
from .decorators import permission_required

@login_required
@permission_required('gestor')
def minha_view_restrita(request):
    # Apenas usuários logados com nível 'gestor' ou superior
    # podem acessar esta view.
    ...

    Este sistema de controle de acesso baseado em papéis (RBAC - Role-Based Access Control) é flexível e permite uma gestão granular das permissões em toda a aplicação.
