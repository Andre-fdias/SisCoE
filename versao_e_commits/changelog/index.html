    Changelog

    O CHANGELOG.md é um arquivo que registra todas as alterações notáveis feitas na aplicação em cada nova versão. Ele serve como um resumo legível por humanos do que mudou, incluindo novas funcionalidades, correções de bugs e breaking changes.

    Geração Automatizada

    Manter um changelog manualmente é um processo tedioso e propenso a erros. Ao adotar o Padrão de Commits, o changelog pode ser gerado e atualizado de forma automática.

    Ferramentas como o standard-version (para projetos Node.js, mas pode ser adaptado) ou o conventional-changelog-cli podem ler o histórico de commits do Git desde a última tag de versão e gerar uma nova seção no CHANGELOG.md com todas as mudanças relevantes.

    Processo de Release

    Um processo de release utilizando a geração automática de changelog seguiria estes passos:

    1. Finalizar o Desenvolvimento: Garantir que todo o código para a nova versão esteja na branch principal (main ou develop).
    2. Executar o Comando de Release: Rodar o comando da ferramenta escolhida. Por exemplo: 
      # Exemplo usando standard-version
npm run release
    3. O que o comando faz:
      • Analisa os commits desde a última tag.
      • Determina a nova versão do projeto (major, minor ou patch) com base nos tipos de commit (feat para minor, fix para patch, BREAKING CHANGE para major).
      • Gera (ou atualiza) o CHANGELOG.md com as novas alterações.
      • Cria um novo commit incluindo o CHANGELOG.md atualizado e a nova versão no package.json (ou outro arquivo de versão).
      • Cria uma nova tag Git com o número da versão (ex: v1.2.0).
    4. Publicar a Release: Fazer o push dos commits e da nova tag para o repositório remoto. 
      git push --follow-tags origin main

    Exemplo de Estrutura do CHANGELOG.md

    Um CHANGELOG.md gerado automaticamente teria a seguinte aparência:

    # Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).
