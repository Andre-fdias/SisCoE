    Padrão de Commits

    Para manter um histórico de commits limpo, legível e significativo, o projeto SisCoE adota a especificação Conventional Commits. Este padrão ajuda a comunicar a natureza das mudanças para outros desenvolvedores e permite a automação de processos, como a geração de changelogs.

    Estrutura de um Commit

    Cada mensagem de commit deve seguir a seguinte estrutura:

    <tipo>[escopo opcional]: <descrição>

[corpo opcional]

[rodapé opcional]

    Tipo

    O tipo deve ser um dos seguintes:

    • feat: Uma nova funcionalidade (feature) para o usuário.
    • fix: Uma correção de bug.
    • docs: Alterações apenas na documentação.
    • style: Alterações que não afetam o significado do código (espaços em branco, formatação, ponto e vírgula, etc).
    • refactor: Uma alteração de código que não corrige um bug nem adiciona uma funcionalidade.
    • perf: Uma alteração de código que melhora a performance.
    • test: Adicionando testes que faltavam ou corrigindo testes existentes.
    • chore: Alterações no processo de build, em arquivos de configuração ou em ferramentas auxiliares. Não altera o código-fonte da aplicação.
    • revert: Reverte um commit anterior.

    Escopo (Opcional)

    O escopo fornece um contexto adicional para o commit. Geralmente, é o nome do módulo ou da parte da aplicação que foi afetada.

    • feat(efetivo): adicionar campo de tipo sanguíneo ao cadastro
    • fix(accounts): corrigir redirecionamento após login
    • docs(docker): atualizar instruções de setup

    Descrição

    A descrição é um resumo conciso das alterações, escrito no imperativo presente.

    • Use "adicionar" em vez de "adicionado" ou "adicionando".
    • Comece com letra minúscula.
    • Não termine com ponto final.

    Corpo (Opcional)

    O corpo do commit é usado para fornecer mais detalhes sobre a alteração, como o contexto da mudança e a motivação para ela.

    Rodapé (Opcional)

    O rodapé é usado para referenciar issues ou Pull Requests. Em particular, para indicar Breaking Changes (mudanças que quebram a compatibilidade).

    • BREAKING CHANGE: o campo 'permissoes' no modelo User agora é obrigatório.

    Exemplos

    Commit de uma nova funcionalidade: 

    feat(agenda): permitir que usuários anexem arquivos a tarefas

Adiciona um modelo `AnexoTarefa` e atualiza a view e o template
para permitir o upload de arquivos associados a uma tarefa.

    Commit de uma correção de bug: 

    fix(efetivo): corrigir cálculo de idade para anos bissextos

O cálculo anterior não levava em conta corretamente os anos
bissextos, resultando em uma idade incorreta por um dia.

Refs: #123

    Commit com uma Breaking Change: 

    refactor(accounts): renomear modelo Profile para UserProfile

BREAKING CHANGE: O modelo `accounts.Profile` foi renomeado para
`accounts.UserProfile`. Todas as importações e referências
precisam ser atualizadas.

    Adotar este padrão tornará o histórico do Git muito mais valioso e fácil de navegar.
