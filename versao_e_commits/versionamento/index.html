    Versionamento Semântico (SemVer)

    O projeto SisCoE adota o Versionamento Semântico (SemVer) para gerenciar os números de versão da aplicação. O SemVer é um conjunto de regras simples que ditam como os números de versão são atribuídos e incrementados.

    A utilização do SemVer, em conjunto com o Padrão de Commits, torna as novas versões do software previsíveis e comunica claramente o escopo das alterações.

    Formato da Versão

    Uma versão SemVer tem o formato MAJOR.MINOR.PATCH.

    • MAJOR (Principal): Incrementada quando são feitas alterações incompatíveis na API (breaking changes).
    • MINOR (Menor): Incrementada quando são adicionadas novas funcionalidades de forma retrocompatível.
    • PATCH (Correção): Incrementada quando são feitas correções de bugs de forma retrocompatível.

    Relação com o Padrão de Commits

    A beleza de usar o SemVer com o Conventional Commits é que a decisão de qual parte da versão incrementar pode ser automatizada:

    • Um commit do tipo feat resulta em um incremento de versão MINOR (1.2.0 -> 1.3.0).
    • Um commit do tipo fix resulta em um incremento de versão PATCH (1.2.0 -> 1.2.1).
    • Um commit que inclui BREAKING CHANGE: no rodapé resulta em um incremento de versão MAJOR (1.2.0 -> 2.0.0).

    Outros tipos de commit (docs, style, refactor, test, chore) não disparam um incremento de versão por si só.

    Exemplo de Fluxo

    1. Versão Atual: 1.5.2
    2. Um desenvolvedor faz um commit para corrigir um bug: 
      fix(accounts): impedir login com usuários inativos
    3. Outro desenvolvedor adiciona uma nova funcionalidade: 
      feat(relatorios): adicionar exportação de efetivo para PDF
    4. Ao executar o processo de release automatizado (descrito na seção Changelog), a ferramenta de conventional-changelog analisará esses commits.
    5. Como há um feat, a nova versão será 1.6.0 (um incremento MINOR). Se houvesse apenas o fix, a nova versão seria 1.5.3 (um incremento PATCH).

    Versão Inicial (0.y.z)

    Enquanto o projeto está em desenvolvimento inicial (versão 0.y.z), qualquer alteração pode resultar em uma breaking change. A versão 1.0.0 marca o primeiro lançamento estável e público da aplicação.

    A adoção do Versionamento Semântico é crucial para uma boa comunicação entre a equipe de desenvolvimento e para a gestão de dependências e releases.
